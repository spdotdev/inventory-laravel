## Tier A ‚Äî real bugs
- [x] **X1 (android) Logout leaks the previous user's data into the next login.** `AuthRepositoryImpl.logout()` does `api.logout(); tokenStore.clear()` and nothing else, but every repository is `@Singleton` with an in-memory cache (Household/Location/Shelf/Product), `HierarchyStore` is a `@Singleton` whose `_state` is never reset, and `FavoritesStore`/`DefaultHouseholdStore` persist. On the login transition `MainActivity` calls `refresh()` ‚Üí `loadFromCache()`, which repopulates User A's households/locations/products/missing-items and favorites for User B until (and, on a failed first refresh, after) the network returns. Fix: add a `clear()` to each repo cache + `HierarchyStore` (`_state.value = HierarchyState()`) + the two prefs stores, and call them all from `logout()` and the 401 path. In-memory only ‚Äî respects always-online/no-DB.
- [x] **X2 (android) Per-location "‚öÝ needs attention" indicator is dead on load.** Home (`AllStoragesScreen`) and the drawer render the warning purely from `state.locationWarnings`, which is populated *only* by `reportLocationWarning` fired from a composed `ProductsPane` ‚Äî so a location whose out-of-stock mandatory item is on a non-visible shelf shows no ‚öÝ until the user opens it and swipes to that shelf. `HierarchyStore.buildFromNetwork` never computes it. Fix: seed `locationWarnings` in `buildFromNetwork` by grouping the `missingItems` it already builds by `locationId` (server-authoritative, no new endpoint); keep the live pane reports as an override for freshly-restocked shelves.
- [x] **X3 (laravel) Password-reset email link is built from `APP_URL`, not `inventory.domain`.** `ForgotPasswordController` does `url(route('inventory.reset-password', ‚Ä¶, absolute: false))`, prefixing the path with `config('app.url')` and discarding the route's own domain. On a supported split-domain deploy (`INVENTORY_DOMAIN` ‚âÝ host `APP_URL`), the emailed link points at the host app, where `/reset-password` isn't registered ‚Üí 404 ‚Üí users can't reset. `HouseholdController::invite()` already uses `config('inventory.domain')` for this reason. Fix: build an absolute URL on the route's domain. Test the link host.
- [x] **X4 (android) Creating/joining/leaving a household never refreshes `HierarchyStore`.** `HouseholdsViewModel.create()/leave()` and `JoinHouseholdViewModel.join()` update only the `HouseholdRepository` cache + their own UI state; the drawer/home/dashboard (driven by `HierarchyStore`) stay stale until a manual pull-to-refresh or relaunch. Product create/delete already call `store.refresh()`; household-level mutations were missed. Fix: inject `HierarchyStore` and call `refresh()` after a successful create/join/leave. Unit-test the refresh is triggered.
- [x] **X5 (laravel) `add()` can push `quantity` past `MAX_QUANTITY` / the unsigned ceiling.** W14 caps per-request `amount` and stored `quantity` (create/update) at 1,000,000, but `add()` uses `increment()` with no ceiling on the resulting total ‚Äî repeated adds from a near-cap quantity can exceed `unsignedInteger` (SQLSTATE 22003 ‚Üí 500) and always violate the "‚â§ 1,000,000" invariant. Fix: clamp atomically (`LEAST(quantity + N, MAX)` via a portable `CASE`, mirroring `remove()`). Test the ceiling holds.

## Tier B ‚Äî CI parity (sibling of a wave-1/2 fix, applied to one repo only)
- [x] **X6 (CI/laravel) `audit.yml` `dependency-review` job lacks `continue-on-error`.** The identical Android job got `continue-on-error: true` under T22 (dependency-graph-not-enabled false-reds); the Laravel one was missed ‚Üí every `composer.json`/`.lock` PR reds on infra. ~~Fix: mirror the Android workflow.~~ **Revised (security review):** mirroring would have made the job *fail-open* ‚Äî a gate that silently passes on real CVEs. Instead **removed the `dependency-review` job entirely**: `composer audit --locked` in the same workflow is already the blocking dependency-CVE gate, and the GitHub-native action needs Dependency Graph / Advanced Security (unavailable here ‚Üí it hard-errors on infra, not on findings). A fail-open gate is worse than no duplicate gate. (The Android `continue-on-error` from T22 should be revisited the same way in a later wave.) **Resolved 2026-07-04:** the Android `dependency-review.yml` job was removed for the same reason ‚Äî but GitHub doesn't populate the Dependency Graph from Gradle build files (no lockfile equivalent to Composer's, so there was no `composer audit`-style backstop). Replaced it with a real, no-CI-infra gate: **Dependabot alerts + automated security fixes enabled on the repo** (Dependabot parses Gradle directly) plus `.github/dependabot.yml` for weekly gradle + github-actions update PRs.
- [x] **X7 (CI/android) `ci.yml` `paths:` omits its own workflow file.** W20 added `.github/workflows/ci.yml` to the Laravel `ci.yml` paths, but the symmetric Android gap remains (`.yml` isn't caught by `**/*.kts`) ‚Üí a PR editing only the Android CI workflow doesn't run the build/test job. Fixed: added `.github/workflows/ci.yml` to both `push` and `pull_request` `paths:` lists.

## Tier C ‚Äî correctness / consistency polish
- [x] **X8 (android) No unit test for the `AuthInterceptor` 401 ‚Üí clear-token session-expiry path.** The whole reactive-logout mechanism (feeds `sessionActive`) is unexercised at unit level, despite CLAUDE.md naming auth/token handling a critical path. Fixed: pure-JVM `AuthInterceptorTest` ‚Äî 401 clears + flips state; 200 leaves it; `Authorization: Bearer` attached only when a token exists. (MockWebServer is `androidTestImplementation`-only, off the JVM unit classpath, so used a fake `Interceptor.Chain` + fake `TokenStore` instead ‚Äî no emulator needed.)
- [x] **X9 (laravel) MCP `SearchUsersTool` still interpolates raw LIKE wildcards.** The exact bug W11 fixed in `AdminController::searchUsers`, left in the MCP tool. Fixed: applied the same `str_replace` + `ESCAPE '!'` escaping to name/email. (`laravel/mcp` is a suggested dep, so `Mcp/` is excluded from the phpstan/test job ‚Äî verified by inspection against the W11 pattern.)
- [x] **X10 (laravel) Household product search is unbounded (no limit).** Unlike admin (`limit(50)`) and MCP (`limit(50)`), `SearchController` returns all matches, and an empty `q` returns the whole catalog. Fixed: added `->limit(50)` consistent with the other two surfaces. Test seeds 60 products, asserts 50 returned.
- [x] **X11 (laravel) CLI `inventory:household:add-member` looks up email case-sensitively.** W13 normalized web auth to lowercase (stored lowercase), but this CLI does not ‚Üí on case-sensitive SQLite, `add-member 1 Foo@x.com` silently adds nobody for a `foo@x.com` user. Fixed: `Str::lower($email)` on lookup. Test adds a member by a mixed-case argument.
- [x] **X12 (android) Google-token 401 surfaces "session expired".** A failed Google sign-in (bad `id_token`) ‚Üí 401. ~~Fix: server `google()` fails with 422.~~ **Revised:** on re-check the client already maps this correctly ‚Äî `AuthViewModel.toGoogleAuthErrorMessage()` maps a Google 401 ‚Üí "Google sign-in failed. Please try again." (its own copy, not the email/password "Incorrect email or password" or a session-expiry string), so no server or client behaviour change was warranted. Instead **added a regression test** pinning the Google-401 copy, so a future refactor can't route the Google path through the shared password mapper. (Made `FakeAuthRepository`'s failure `Throwable` configurable to inject an `HttpException(401)`.)
